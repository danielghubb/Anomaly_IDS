# DFIR-IRIS Usage Guide

## Overview

DFIR-IRIS is an open-source incident response and case management platform.

Key IRIS components in this setup:
- **`iris`**: Nginx reverse proxy serving the IRIS web UI (on `10.0.0.6` inside the `island` network).
- **`iris_app`**: IRIS application backend (on `10.0.0.7` inside the `island` network).
- **`iris_db`**: PostgreSQL database for IRIS.
- **`iris_rabbitmq`**: Message broker used by IRIS `app` and `worker`.

This setup is **lab-only** and uses **hardcoded admin credentials and API token** for convenience.


## How to Access the DFIR-IRIS Web UI

### Network locations

Inside the Docker `island` network:
- `iris` (Nginx): `10.0.0.6` or `iris`
- `iris_app`: `10.0.0.7` or `iris_app`

On the Host machine you access IRIS with:

```text
https://localhost
```

### certificate warning

IRIS uses self-signed certificates in this test setup.  
Your browser will likely show a certificate warning. 
For `curl` commands use `-k`. 

In case the certificate issue results in problems, the iris Root CA cert can be found in `infrastructure/iris-web/certificates/rootCA/irisRootCACert.pem`

## Usage
A small explenation of the DFIR-IRIS features.
### Alerts
**Alerts** are automated notifications generated by detection systems (Suricata, ML models, IDS/IPS).

**Example**: Suricata generates 100 alerts about port scanning from `10.0.0.10`.

### Cases
**Cases** are investigation containers managed by analysts in DFIR-IRIS.

This should be the location where we can fullfil the in depth data from the project task. Solve the problem :"One drawback is the usefulness of alarms for operational purposes."

### The Relationship
```
[Many Alerts]  → [One Case]
```

 **Alerts** tell you "something happened".
 
 **Cases** let you investigate "what it means and what to do about it"

In DFIR-IRIS, alerts are typically represented as **Events** within a case.
